# 技術選定・アーキテクチャ決定ログ

> プロジェクト: Markdown / HTML Editor - Typora ライク WYSIWYG エディタ
> バージョン: 1.0
> 更新日: 2026-02-26

本ドキュメントは「何を採用したか」ではなく「なぜ採用したか / 何を採用しなかったか」を記録する決定ログ。
設計ドキュメント本体（`system-design.md` 等）はこのドキュメントへの参照を記載し、不採用理由の詳述は含まない。

---

## 1. プラットフォーム: Tauri 2.0 を採用・Electron を不採用とした理由

**決定日**: プロジェクト開始時（2026年以前）
**決定**: デスクトップフレームワークとして **Tauri 2.0**（Rust）を採用し、Electron は採用しない。

### 理由

| 観点 | Electron | Tauri 2.0 |
|------|----------|-----------|
| モバイル対応 | **不可**（Chromium ベースのデスクトップ専用） | **可**（Android / iOS を同一コードベースで対応） |
| バイナリサイズ | 大（Chromium を同梱するため 100MB 超） | 小（OS の WebView を使用、数MB〜） |
| メモリ使用量 | 高（Chromium プロセスを含む） | 低（OS の WebView を使用） |
| バックエンド言語 | Node.js（JavaScript） | Rust（型安全・高パフォーマンス） |

本プロジェクトは将来の Android / iOS 展開（roadmap.md Phase 8）を計画しており、Electron はこの要件を満たせない。

## 2. エディタエンジン: TipTap を採用・ProseMirror 直接使用を不採用とした理由

**決定日**: プロジェクト開始時
**決定**: エディタエンジンとして **TipTap**（ProseMirror ラッパー）を採用し、ProseMirror を直接使用する方式は採用しない。

### 理由

ProseMirror を直接使用する場合、テーブル・数式・コードブロック等を全て自前実装する必要があり、個人開発では工数が過大になる。

TipTap は ProseMirror の**完全なラッパー**であり、以下の利点がある:
- 公式拡張（テーブル・数式・コードブロック・画像等）を活用することで開発速度を維持しつつ、Typora を超える品質を現実的に達成できる
- 内部は ProseMirror そのものであるため、カスタム拡張の余地も十分ある
- TypeScript ネイティブで React との統合が良好
- カーソル / セレクション管理・Undo/Redo は ProseMirror の機構をそのまま継承

## 3. 全文検索エンジン: Rust 内製（walkdir + regex）を採用・ripgrep subprocess を不採用とした理由

**決定日**: search-design.md 設計時
**決定**: ワークスペース横断全文検索のバックエンドは **Rust 内製（walkdir + regex クレート）** で実装し、外部 ripgrep バイナリへの依存は採用しない。

### 理由

| 観点 | ripgrep subprocess | Rust 内製（walkdir + regex） |
|------|-------------------|--------------------------|
| バイナリサイズ | ripgrep を同梱するとサイズが増加 | Cargo.toml に 2 クレート追加のみ |
| クロスコンパイル | 全プラットフォーム向けの ripgrep 配布が複雑 | Rust クレートとして統一ビルド |
| パフォーマンス | 高速 | Rust ネイティブ regex でも JavaScript 比 5〜20 倍高速（十分） |
| 依存関係 | 外部バイナリ依存（バージョン管理が必要） | シンプル |

将来的にワークスペースが数万ファイル規模になった場合は ripgrep ベースへの切り替えを再検討する（`search-design.md §3.2` 参照）。

## 4. フロントエンド構成: レイヤー分離型を採用・機能駆動型を不採用とした理由

**決定日**: system-design.md §3.1 設計時
**決定**: `src/` のディレクトリ構成は **レイヤー分離型**（`core/`, `renderer/`, `components/`）を採用し、機能駆動型パッケージング（`features/editor/`, `features/workspace/`）は採用しない。

### 理由

- `core/` 層の単体テスト（ラウンドトリップテスト）を UI コンポーネントから完全に分離でき、テスト容易性が高い
- `renderer/` 層は TipTap/ProseMirror に強く依存するため、React コンポーネント層と技術的に異なる
- `CLAUDE.md` の `src/` ↔ `docs/` マッピングとの整合性を維持できる
